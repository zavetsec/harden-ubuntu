ZavetSec-Harden-Ubuntu v1.5-ubuntu

Однофайловый инструмент харденинга и compliance-аудита для Ubuntu 20.04 / 22.04 / 24.04. Один Bash-скрипт, ноль зависимостей: детектирует хост, подбирает профиль под роль, проверяет соответствие базлайну или приводит систему к нему — с бэкапом каждого изменения и откатом одной командой.

01 ЧТО ЭТО И ЗАЧЕМ

Большинство скриптов харденинга пытаются поддерживать все дистрибутивы сразу, обрастают непротестированными ветками совместимости и в итоге им страшно давать root. Этот инструмент устроен наоборот: только Ubuntu, с учётом роли машины, сначала аудит — потом изменения.

Скрипт знает release-специфичные ловушки Ubuntu и обходит их автоматически:

  • алгоритм yescrypt существует только с 22.04 — на 20.04 используется SHA512 с настроенными rounds;
  • pam_faillock требует PAM ≥ 1.4 — на 20.04 его физически нет, и скрипт отказывается ломать PAM-стек;
  • sshd 8.2 на focal не знает опцию KbdInteractiveAuthentication — используется старое имя;
  • cloud-образы содержат 50-cloud-init.conf, который перебивает неаккуратно названные drop-in'ы — наш конфиг называется 00-… и выигрывает при семантике «первое совпадение побеждает»;
  • штатный ufw на noble при каждом старте активно сбрасывает log_martians=0 — скрипт правит его собственный sysctl-файл по месту.

Каждый релиз прогоняется на реальных виртуальных машинах через полный цикл apply → reboot → audit → rollback → apply. Именно так были найдены и исправлены баги, которые никогда не поймал бы статический анализ.

ЗАПУСК БЕЗ ROOT

Детект и частичный аудит работают без root, но проверки sshd/ufw/AppArmor уходят в SKIP, и оба формата отчёта выводят заметное предупреждение «NON-ROOT AUDIT — SCORE IS UNRELIABLE», чтобы неполный балл никого не ввёл в заблуждение.

03 ЖИЗНЕННЫЙ ЦИКЛ ЗАПУСКА

детект хоста ──▶ выбор профиля ──▶ аудит (read-only, отчёт со score) │ ▼ apply (бэкап каждого изменения) │ ▼ rollback (одна команда, полный откат прогона)

Детект определяет: дистрибутив и версию (20.04/22.04/24.04), init-систему, признак контейнера, firewall-бэкенд и его активность (SIGPIPE-безопасно), MAC (AppArmor), наличие GUI, живые SSH-порты (из конфига и из ss, с корректным разбором IPv6-адресов), socket-активацию ssh.socket на 24.04 и роль машины. Роль выводится автоматически: контейнер → container-host; есть GUI → workstation; работает dockerd/kubelet → сервер повышается до container-host; иначе → server. Роль можно переопределить флагом --role, профиль — флагом --profile.

ПОЧЕМУ ТАК

Docker публикует порты собственными nftables-правилами в обход ufw — поэтому на container-host модуль firewall намеренно выключен, а скрипт предупреждает об этом. MaxSessions 4 вне серверов — тоже намеренно: значение 2 ломает SSH-мультиплексирование (ControlMaster) и VS Code Remote.

ГАРАНТИИ

Перед перезагрузкой конфига выполняется sshd -t; при провале восстанавливается прежний drop-in из бэкапа, а не просто удаляется наш. Парольная аутентификация отключается только если у root или пользователей найдены authorized_keys, и только по opt-in ZS_TUNE_SSH_DISABLE_PASSWORD=1.

OPT-IN ВКЛЮЧЕНИЕ FAILLOCK В PAM HIGH

Само вплетение faillock в аутентификацию (ZS_TUNE_ENABLE_FAILLOCK_PAM=1) выключено по умолчанию — это риск локаута. При включении используется родной pam-auth-update и канонический стек preauth(1025) / authfail(0) / authsucc(Additional): залоченный отсекается до проверки пароля, неудача считается после pam_unix, успех сбрасывает счётчик. На 20.04, где pam_faillock отсутствует, скрипт отказывается включать wiring вместо записи нерабочего PAM-стека. Держите второй root-сеанс открытым и проверяйте с отдельной сессии.
5FIREWALLufw default-deny

Порядок анти-локаутный: сначала allow на каждый обнаруженный SSH-порт (из конфига и из живых сокетов), затем default-deny входящих, allow исходящих, логирование — и только потом ufw --force enable. Loopback и established-соединения ufw сохраняет сам.

  • если ufw был активен до запуска — его правила снапшотятся в бэкап, и rollback вернёт именно их, а не выключит firewall;
  • активные строки log_martians=0, которые noble шипит в /etc/ufw/sysctl.conf, исправляются по месту (append в конец не помогает — ufw применяет этот файл при каждом старте);
  • после ufw enable повторно выполняется sysctl --system, чтобы значения из /etc/sysctl.d победили сразу, а не после ребута.

DOCKER ВАЖНО

Docker публикует порты собственными nftables-цепочками в обход ufw — на container-host модуль выключен профилем, фильтруйте на уровне оркестратора или cloud security groups.

06 МЕХАНИКА БЕЗОПАСНОСТИ

  • Dry-run по умолчанию. Без --apply система не меняется, а вывод показывает каждое действие — включая команды ufw и systemctl (строки would run:).
  • Бэкап перед каждым изменением. Оригиналы копируются в state-dir/backup с сохранением путей. Аудит и dry-run не создают ни state dir, ни пустых rollback-файлов — вообще ничего.
  • Идемпотентность с верификацией. Повторный запуск безопасен (сплошные already); каждая правка файла проверяется после записи — неудача даёт громкий ERR, а не ложный OK. Закомментированные строки (включая документацию в login.defs) никогда не переписываются.
  • Подтверждение apply. На интерактивном терминале запрашивается явное YES; в CI/автоматизации подтверждение пропускается.
  • Не-Ubuntu отклоняется без флага --force; деривативы (Mint, Pop!_OS) допускаются с предупреждением.

ПЕРВЫЙ УДАЛЁННЫЙ APPLY

Всегда держите второй root-сеанс открытым при первом --apply на удалённой машине — несмотря на все гарантии. Проверка нового SSH-подключения сразу после применения обязательна.

07 АУДИТ И ОТЧЁТЫ

Аудит читает живое состояние системы, а не только конфиги: ядро через sysctl -n, эффективный конфиг SSH через sshd -T (то, что демон реально применил после всех drop-in'ов), ufw status, aa-status, auditctl -l, опции монтирования через findmnt, реальные права через stat, состояние сервисов и пакетов.

Каждая проверка получает статус и важность:

PASS соответствует базлайну   FAIL расхождение — apply исправит   WARN требует внимания, но не однозначный дефект   SKIP неприменимо/нет прав — с причиной в note

HIGH критично для безопасности   MED важно   LOW желательно

Score = PASS / (PASS + FAIL). Отчёты пишутся в текущий каталог (--report-dir меняет) в двух форматах (--format txt|html|both):

  • TXT — по категориям, итоговая строка, явный список провалов с готовой командой исправления. Удобен для логов, cron и diff между прогонами.
  • HTML — самодостаточная страница: шкала compliance, счётчики PASS/FAIL/WARN/SKIP, красная панель провалов «ожидалось vs есть», таблицы по категориям с бейджами важности, живой фильтр-поиск. Ни одного внешнего запроса — отчёт аудита не имеет права «звонить домой».

Контекст отражается честно: на workstation, где apport оставлен намеренно, fs.suid_dumpable показывается как SKIP «apport-managed», а не ложный FAIL.

08 ROLLBACK — ЧТО ОТКАТЫВАЕТ И ЧТО НЕТ

Каждый --apply создаёт каталог /var/log/zavetsec-harden/<метка>/ с бэкапами, журналом, списком изменений и скриптом отката:

sudo bash /var/log/zavetsec-harden/<метка>/rollback.sh

Он восстанавливает файлы из бэкапа, включает обратно отключённые/замаскированные сервисы, откатывает изменения ufw/sysctl/PAM этого прогона, затем выполняет sysctl --system и печатает честное резюме. Границы отката — скрипт сам о них сообщает:

  • Rollback — per-run. Каждый rollback.sh откатывает ровно изменения своего apply. Применяли несколько раз — запускайте содержательные rollback'и от нового к старому.
  • Runtime живёт до ребута. Удаление sysctl-файла не сбрасывает значения в ядре, загруженные audit-правила остаются загруженными — для полного отката нужен reboot.
  • Пакеты остаются намеренно. auditd, unattended-upgrades, sysstat не удаляются (удалять средства безопасности за спиной — хуже); исключение — fail2ban, который убирается, если его ставил именно откатываемый прогон.
  • Прежний ufw восстанавливается, а не выключается. Если firewall был активен до запуска, откат вернёт ваши прежние правила.

10 ЧЕГО СКРИПТ СОЗНАТЕЛЬНО НЕ ДЕЛАЕТ

  • не переписывает /etc/fstab для /tmp и /var/tmp автоматически (только рекомендации);
  • не переводит AppArmor-профили в enforce вслепую;
  • не отключает user namespaces и не трогает snap/squashfs;
  • не ставит GRUB-пароль (модель угроз физического доступа — решение по месту);
  • не перекраивает разделы диска (отдельные /home, /tmp, /var — решение времени установки);
  • не настраивает удалённую отправку логов (специфично для среды);
  • не ставит антивирус и AIDE (тяжеловесно; возможны как явные opt-in позже);
  • не заменяет сертифицированные сканеры — для формальной отчётности пара с Lynis/OpenSCAP.

Границы названы явно: когда Lynis предлагает что-то из списка выше — это задуманное разделение труда, а не пробел. На стоковом десктопе 24.04 применение базлайна поднимает Lynis hardening index примерно с 60 до середины 80-х; остаток — как раз пункты этого списка.

11 ШПАРГАЛКА КОМАНД

# Что скрипт видит на этой машине
sudo ./zavetsec-harden-ubuntu.sh --detect-only

# Аудит соответствия (read-only) → TXT + HTML отчёты
sudo ./zavetsec-harden-ubuntu.sh --audit

# Посмотреть, что будет сделано, ничего не меняя (режим по умолчанию)
sudo ./zavetsec-harden-ubuntu.sh

# Применить базлайн (профиль подбирается по роли автоматически)
sudo ./zavetsec-harden-ubuntu.sh --apply

# Конкретный профиль / роль
sudo ./zavetsec-harden-ubuntu.sh --apply --profile server
sudo ./zavetsec-harden-ubuntu.sh --apply --role container-host

# Только выбранные модули / без выбранных
sudo ./zavetsec-harden-ubuntu.sh --apply --only ssh,firewall
sudo ./zavetsec-harden-ubuntu.sh --apply --skip firewall

# Аудит в каталог, только HTML
sudo ./zavetsec-harden-ubuntu.sh --audit --report-dir /var/log/zs-audit --format html

# Откат последнего применения
sudo bash /var/log/zavetsec-harden/<метка>/rollback.sh
