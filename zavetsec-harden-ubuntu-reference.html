ZAVETSEC // DEFENSIVE TOOLING

ZavetSec-Harden-Ubuntu v1.5-ubuntu

Single-file security hardening and compliance auditing for Ubuntu 20.04 / 22.04 / 24.04. One Bash script, zero dependencies: it detects the host, selects a role-appropriate profile, then either measures compliance against the baseline or enforces it — backing up every change and providing a one-command rollback.

01 WHAT IT IS AND WHY

Most Linux hardening scripts try to support every distribution, accumulate years of compatibility branches nobody re-tests, and eventually become hard to trust with root. This tool is built the opposite way: Ubuntu only, role-aware, audit-first.

It knows Ubuntu's release-specific traps and handles them automatically:

  • yescrypt exists only from 22.04 — on 20.04 the tool falls back to SHA512 with tuned rounds;
  • pam_faillock requires PAM ≥ 1.4 — it simply doesn't exist on 20.04, and the tool refuses to write a broken PAM stack there;
  • sshd 8.2 on focal rejects the modern KbdInteractiveAuthentication option name — the legacy name is used instead;
  • cloud images ship a 50-cloud-init.conf that silently overrides carelessly named drop-ins — our config is named 00-… and wins under sshd's first-match-wins semantics;
  • stock ufw on noble actively resets log_martians=0 on every start — the tool fixes ufw's own sysctl file in place.

Every release is exercised on real VMs through the full apply → reboot → audit → rollback → apply cycle. That process caught — and fixed — bugs no static review would ever find.

RUNNING WITHOUT ROOT

Detection and a partial audit work without root, but the sshd/ufw/AppArmor checks SKIP, and both report formats display a prominent “NON-ROOT AUDIT — SCORE IS UNRELIABLE” warning so an incomplete score never misleads anyone.

03 RUN LIFECYCLE

detect host ──▶ select profile ──▶ audit (read-only, scored report) │ ▼ apply (backup every change) │ ▼ rollback (one command, full revert of the run)

Detection covers: distribution and release (20.04/22.04/24.04), init system, container status, firewall backend and whether it's active (SIGPIPE-safe), MAC (AppArmor), GUI presence, live SSH port(s) — taken from both the config and ss, with IPv6-safe address parsing — plus ssh.socket activation on 24.04, and the machine's role. The role is inferred automatically: container → container-host; GUI present → workstation; dockerd/kubelet running → server is promoted to container-host; otherwise → server. Override the role with --role, the profile with --profile.

WHY IT'S BUILT THIS WAY

Docker publishes ports through its own nftables chains that bypass ufw — so the firewall module is intentionally off on container-host, and the tool warns about it. MaxSessions 4 outside servers is also deliberate: a value of 2 breaks SSH multiplexing (ControlMaster) and VS Code Remote.

Eleven modules in run order. Any subset can be run (--only ssh,firewall) or excluded (--skip firewall). Every enforcement module has an audit mirror that shares the same baseline — an audit's FAIL list is precisely what apply will change.

GUARANTEES

sshd -t runs before any reload; on failure the previous drop-in is restored from backup — never just deleted. Password authentication is only ever disabled when authorized_keys exist for root or users, and only behind the explicit opt-in ZS_TUNE_SSH_DISABLE_PASSWORD=1. sshd is reloaded, never killed, and ssh.socket activation on 24.04 is handled correctly.

OPT-IN WIRING FAILLOCK INTO PAM HIGH

Actually wiring faillock into authentication (ZS_TUNE_ENABLE_FAILLOCK_PAM=1) is off by default — it's a lockout risk. When enabled, the native pam-auth-update mechanism is used with the canonical preauth(1025) / authfail(0) / authsucc(Additional) stack: a locked user is refused before the password check, a failure is counted after pam_unix, and a success resets the counter. On 20.04, where pam_faillock doesn't exist, the tool refuses to enable the wiring rather than write a broken PAM stack. Keep a second root session open and verify from a separate session.
5FIREWALLufw default-deny

The order is anti-lockout by design: first allow for every detected SSH port (from the config and from live sockets), then default-deny inbound, allow outbound, logging — and only then ufw --force enable. Loopback and established connections are preserved by ufw itself.

  • if ufw was active before the run, its ruleset is snapshotted into the backup dir — rollback restores those rules instead of disabling the firewall;
  • the active log_martians=0 lines noble ships in /etc/ufw/sysctl.conf are fixed in place (appending =1 at the end isn't enough — ufw applies that file on every start);
  • after ufw enable, sysctl --system is re-run so the values from /etc/sysctl.d win immediately, not just at the next boot.

DOCKER IMPORTANT

Docker publishes container ports through its own nftables chains that bypass ufw — the module is disabled by the container-host profile; filter at the orchestrator or cloud-security-group layer.

06 SAFETY MACHINERY

  • Dry-run by default. Without --apply the system is untouched, and the output shows every intended action — including ufw and systemctl commands (the would run: lines).
  • Backup before every change. Originals are copied into state-dir/backup, preserving paths. Audit and dry-run create no state directory, no empty rollback files — nothing at all.
  • Idempotent and verified. Re-running is safe (a wall of already); every in-place file edit is verified after writing — a failure produces a loud ERR, never a false OK. Commented lines (including the documentation comments in login.defs) are never rewritten.
  • Apply confirmation. On an interactive terminal an explicit YES is required; in CI/automation the prompt is skipped.
  • Non-Ubuntu is refused without --force; derivatives (Mint, Pop!_OS) are detected and allowed with a warning.

FIRST REMOTE APPLY

Always keep a second root session open the first time you run --apply on a remote machine — guarantees or not. Testing a fresh SSH connection immediately after applying is mandatory.

07 AUDIT AND REPORTS

The audit reads the live system state, not just config files: the kernel via sysctl -n, the effective SSH config via sshd -T (what the daemon actually resolved after all drop-ins), ufw status, aa-status, auditctl -l, mount options via findmnt, real permissions via stat, service and package state.

Every check gets a status and a severity:

PASS matches the baseline   FAIL a deviation — apply will fix it   WARN needs attention, not necessarily a defect   SKIP not applicable / insufficient privileges — reason in the note

HIGH security-critical   MED important   LOW desirable

Score = PASS / (PASS + FAIL). Reports land in the current directory (--report-dir to change) in two formats (--format txt|html|both):

  • TXT — grouped by category, a summary line, and an explicit failure list with the exact remediation command. Great for logs, cron jobs and diffing between runs.
  • HTML — a self-contained page: compliance gauge, PASS/FAIL/WARN/SKIP tallies, a red panel listing every failure as “expected vs got”, per-category tables with severity badges, and a live filter box. Not a single external request — an audit artifact has no business phoning home.

Context is reported honestly: on workstation, where apport is deliberately kept, fs.suid_dumpable shows as SKIP “apport-managed” instead of a false FAIL.

08 ROLLBACK — WHAT IT REVERTS AND WHAT IT CAN'T

Every --apply creates /var/log/zavetsec-harden/<timestamp>/ with backups, a log, the change list and the rollback script:

sudo bash /var/log/zavetsec-harden/<timestamp>/rollback.sh

It restores files from backup, re-enables disabled/masked services, undoes the run's ufw/sysctl/PAM changes, then runs sysctl --system and prints an honest summary. The boundaries — the script itself tells you all of this when it finishes:

  • Rollback is per-run. Each rollback.sh reverts exactly what its apply changed. Applied several times? Run each content-bearing rollback from newest to oldest.
  • Runtime state persists until reboot. Deleting a sysctl file doesn't reset the values inside the kernel, and loaded audit rules stay loaded — a full revert needs a reboot.
  • Packages stay by design. auditd, unattended-upgrades, sysstat are not removed (uninstalling security tooling behind your back would be worse); the exception is fail2ban, removed if — and only if — the run being rolled back installed it.
  • A pre-existing ufw ruleset is restored, not disabled. If the firewall was active before the run, rollback puts your previous rules back.

10 WHAT IT DELIBERATELY DOES NOT DO

  • doesn't rewrite /etc/fstab for /tmp and /var/tmp automatically (advisory only);
  • doesn't flip AppArmor profiles to enforce blindly;
  • doesn't disable user namespaces and doesn't touch snap/squashfs;
  • doesn't set a GRUB password (a physical-access threat model — decide per site);
  • doesn't repartition disks (separate /home, /tmp, /var are install-time decisions);
  • doesn't configure remote log shipping (environment-specific by nature);
  • doesn't install a malware scanner or AIDE (heavyweight; may appear later as explicit opt-ins);
  • doesn't replace certified scanners — pair with Lynis/OpenSCAP for formal reporting.

The boundaries are named on purpose: when Lynis suggests something from the list above, that's the intended division of labour, not a gap. On a stock 24.04 desktop, applying the baseline typically moves the Lynis hardening index from roughly 60 into the mid-80s; the remainder is exactly this list.
